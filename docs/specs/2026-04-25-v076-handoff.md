# Tripwire v0.7.6 ‚Äî handoff spec

**Status**: spec (ready for plan)
**Date**: 2026-04-25
**Depends on**: v0.7.5 (`2026-04-24-v075-handoff.md`) ‚Äî draft PRs at
session-start, spawn template tightening, review-feedback cycle,
pm-review.md artifact.
**Sibling**: v0.8 active concept graph
(`2026-04-24-v08-bidirectional-concept-graph.md`) ‚Äî independent
track; v0.7.6 must not pull v0.8 work forward.
**Source**: gaps surfaced while standing up the
`tripwire-v0` project-tracking repo (renamed from
`project-tripwire-ui-init` 2026-04-25). Five concrete failures every
new project hits or would hit:

1. `tripwire init` doesn't create the GitHub repo or set up the
   remote ‚Äî operator has to do it by hand. Without a remote, v0.7.4's
   per-session project-tracking worktrees never activate, and v0.7.5's
   draft PRs hit the graceful-skip path. Both features are
   dead-on-arrival for new projects.
2. `tripwire init` stamps a CI workflow + canonical enums into a new
   project, but there's no command to **re-stamp** when those
   templates evolve. The `tripwire-v0` project is missing the CI
   workflow entirely (init was run before the template existed) and
   has a stale `enums/session_status.yaml` lacking `executing` /
   `done` (the new tripwire vocabulary). Manual fix per-project
   doesn't scale.
3. The init template's `enums/session_status.yaml` lists the legacy
   state machine (`active`, `waiting_for_ci`, ‚Ä¶) instead of what the
   tripwire CLI now emits (`executing`, `done`). Every newly-init'd
   project will hit `enum/session_status` validation errors the moment
   a session runs.
4. `lint/issue_body_orphan_concepts` is a regex+stopword heuristic
   that produces ~2,251 warnings on a normal-sized project (every
   `## Context`, `## Implements`, `## Done` markdown header is
   flagged). It exits non-zero on warnings, so CI fails on every PR
   for noise the operator can't reasonably address. The signal is
   actively training operators to ignore lint output, which corrodes
   every other rule.
5. CI workflows in the `tripwire` code repo and the project-init
   template are pinned to deprecated GitHub Action majors
   (`actions/checkout@v4`, `actions/setup-node@v4`,
   `astral-sh/setup-uv@v3`). Node 20 is being removed from runners on
   2026-09-16. setup-uv@v3 also rejects `python-version` per a
   schema change, so the workflow as shipped doesn't work cleanly
   anyway.

v0.7.6 closes all five with five items. Theme: **project plumbing &
hygiene** ‚Äî make `tripwire init` produce a working project end to
end, add a maintenance command (`migrate`) to retrofit existing
projects, replace the noisy lint rule with a precise one, and
document the CI-version bump as the canonical pattern.

## 2. Items

### A. `tripwire init` auto-creates the GitHub repo + configures the remote

(Moved verbatim from v0.7.5 ¬ß2.F ‚Äî see `2026-04-24-v075-handoff.md`
prior revision for the original framing. Recapped here as the
canonical v0.7.6 description.)

**Problem.** v0.7.4 ships per-session project-tracking worktrees and
v0.7.5 ships draft PRs from session-start. Both need a remote on the
project-tracking repo. Today `tripwire init` does `git init` and
stops ‚Äî no remote, no auto-push. Both features fall through to
graceful-skip / manual fallback paths until the operator runs `git
remote add origin ‚Ä¶` and pushes the initial commit. Worse: even
after adding a remote, the first session spawn fails on `git push -u
origin proj/<sid>` if upstream main was never pushed.

**Solution.** `tripwire init` defaults to creating the GitHub repo,
adding the remote, and pushing the initial commit. No extra flags
needed for the happy path. Configurable opt-outs for local-only /
experimental / pre-existing-repo / network-flaky cases.

**No `gh` dependency.** Tripwire calls the GitHub REST API directly
via `httpx` (already a transitive dependency through FastAPI's
`TestClient`). Auth via `GITHUB_TOKEN` or `GH_TOKEN` env var or
reading `~/.config/gh/hosts.yml`. Gives finer-grained error handling
than parsing `gh` exit codes.

#### A1. New init flow

After the existing repos prompts (v0.7.3 I2), `tripwire init`:

1. **Resolve the GitHub target.** Prompt for owner + name with a
   sensible default (`<owner>/<project_dir.name>`). In
   `--non-interactive` mode the default is used unless overridden
   via `--github-owner <owner>` / `--github-repo <name>`.
2. **Check whether the repo already exists** via `GET /repos/{owner}/{name}`.
   - 200 ‚Üí existing repo. Skip create, just add the remote pointing
     at the SSH URL. Log the SSH URL in the init summary.
   - 404 ‚Üí create.
   - 401/403 ‚Üí token missing or unauthorised. Fail-fast with a clear
     "set GITHUB_TOKEN / GH_TOKEN" error.
3. **Create the repo** via `POST /user/repos` (personal namespace) or
   `POST /orgs/{owner}/repos` (org namespace, decided by whether
   `owner` matches the authenticated user). Body: `{name, private:
   true, auto_init: false, description: "Tripwire project-tracking
   repo for <name>"}`. `private` default ‚Äî operator opts out via
   `--public` for OSS visibility.
4. **Add the remote.** `git remote add origin
   git@github.com:<owner>/<name>.git` (SSH; same convention as code
   repos).
5. **Push the initial commit.** `git push -u origin main`. Makes the
   remote "hot" so every subsequent session spawn can assume
   `origin/main` exists.

#### A2. Opt-out flags

Three independent escape hatches:

- `--no-github-repo` ‚Äî skip steps 2-3. Step 4 still adds the remote
  URL the operator types in; step 5 still pushes. For operators who
  pre-create repos or use Terraform.
- `--no-remote` ‚Äî skip steps 2-5 entirely. Pre-v0.7.6 behaviour.
  Local-only / experimental / sandbox.
- `--no-push` ‚Äî keep steps 2-4, skip step 5. Remote configured but
  cold (no upstream pushed). Useful when network is flaky or the
  operator wants to inspect the repo first.

#### A3. Token resolution

`GITHUB_TOKEN` first, then `GH_TOKEN`, then read from
`~/.config/gh/hosts.yml` (`gh auth login` writes it; even though we
don't shell out to `gh`, reading its config is a UX win). Fail-fast
with one clear message if all three miss.

#### A4. Schema migration on `project.yaml`

Add an optional top-level `project_repo_url` field recording the
GitHub URL of the project-tracking repo itself. Useful for the UI's
"open in GitHub" button and disambiguates from `repos:` (code repos).
Backwards-compatible ‚Äî absence means pre-v0.7.6 init.

#### A5. Code

- `src/tripwire/core/github_client.py` ‚Äî NEW. Thin httpx wrapper:
  `repo_exists(owner, repo) -> bool`, `create_repo(owner, repo, *,
  private, description) -> dict`, `resolve_token() -> str | None`,
  `_authenticated_owner() -> str | None`. ~80 lines.
- `src/tripwire/cli/init.py` ‚Äî extend `_prompt_for_repo_locals`'s
  call site with `_prompt_for_github_remote(...)` returning
  `(owner, name, push)` or `None`. New helper
  `_setup_github_remote(...)` orchestrates A1 steps 2-5.
- `src/tripwire/models/project.py` ‚Äî add optional
  `project_repo_url: str | None = None`.
- `pyproject.toml` ‚Äî add `httpx` to `dependencies` (already pulled
  in transitively via `fastapi[testclient]`; declaring explicitly is
  cleaner).

#### A6. Tests

- `tests/unit/test_github_client.py` ‚Äî NEW: `test_repo_exists_returns_true_on_200`,
  `test_repo_exists_returns_false_on_404`,
  `test_repo_exists_raises_on_401`,
  `test_create_repo_calls_user_endpoint_for_self_owned`,

### B. Canonical session-status enum in init template

**Problem.** `src/tripwire/templates/project/enums/session_status.yaml`
ships values from the legacy state machine (`active`,
`waiting_for_ci`, `waiting_for_review`, `waiting_for_deploy`,
`re_engaged`, `completed`, `failed`). The tripwire CLI's session
lifecycle now uses `planned`, `queued`, `executing`, `paused`,
`failed`, `completed`, `done` (with `executing`/`done` being the
day-to-day in-flight + post-complete states). Every newly-init'd
project hits `enum/session_status not in active session_status enum`
validation errors the moment a session is spawned.

The `tripwire-v0` project hit exactly this ‚Äî its enum had to be
hand-edited to add `executing` + `done` before validate exited 0.

### C. `tripwire migrate` ‚Äî re-stamp missing template files

**Problem.** `tripwire init` stamps a fixed set of files into a new
project (CI workflow, enums, manifests, skill bundle). When those
templates evolve, existing projects don't get the update. There's no
canonical command to "re-stamp anything missing or stale". Operators
either notice and fix manually (which is what just happened with the
CI workflow on `tripwire-v0`) or ‚Äî more often ‚Äî never notice.

**Solution.** A new `tripwire migrate` command that re-runs the
template-stamping logic idempotently:

- Walks every file `tripwire init` would write today.
- For each: if absent in the project, write it. If present, **leave
  it** (no overwrite). Print a summary diff: "X files added, Y files
  unchanged, Z files differ from template (use `--show-diffs` to see
  them)".
- `--show-diffs` prints the full unified diff per Z file but doesn't
  modify them. Lets the operator decide which deltas are intentional
  customisation vs drift.
- `--force=<path1>,<path2>` selectively overwrites the listed
  template-managed files. No `--force-all` ‚Äî too easy to clobber
  customisation.

#### C1. Scope (template re-stamping ONLY)

In scope:
- `.github/workflows/tripwire.yml` (CI)
- `enums/*.yaml` (canonical enums ‚Äî but only ADD; never change
  existing values, since downstream data references them)
- `templates/artifacts/manifest.yaml` and the `*.j2` artifact
  templates
- The `.claude/skills/project-manager/` skill bundle

Out of scope (explicit non-goals):
- Session.yaml schema migrations between tripwire versions (much
  bigger; needs a `_schema_version` field first; v0.8+).
- Issue body migrations (e.g. mass renaming a frontmatter field).
- `project.yaml` schema migrations.
- Re-running `tripwire init` interactively.

#### C2. Idempotency contract

Re-running `tripwire migrate` twice in a row produces zero changes
the second time. Tests assert this. Implementation: stamp into a
temporary dir first, then diff against the project; only write files
that are absent.

#### C3. Drift detection

When a file exists but its content differs from the current
template, the file is reported as "drifted" (not modified). The
operator's options:
- Accept drift: do nothing. The migrate command prints the diff if
  asked but never modifies.
- Reset to template: `tripwire migrate --force <path>`.
- Update the template: edit the canonical template in tripwire's own
  repo (separate workflow).

This avoids the `tripwire init --upgrade` failure mode where the
"upgrade" silently overwrites customisation.

#### C4. CLI surface

```
tripwire migrate [--project-dir <dir>] [--show-diffs] [--force <path>...]
```

Output (text default; `--format json` opts in to machine-parseable):

```
Migrating /Users/maia/Code/seido/projects/tripwire-v0 ‚Ä¶

Added (3):
  .github/workflows/tripwire.yml
  templates/artifacts/pm-review.md.j2
  .claude/skills/project-manager/references/WORKFLOWS_REVIEW.md (delta vs old)

Unchanged (7):
  templates/artifacts/manifest.yaml
  enums/session_status.yaml  ‚Üê already up-to-date
  ‚Ä¶

Drift (1):
  enums/issue_status.yaml
    ‚Üí 2 values added in template: ('blocked', 'verifying'). Use
      --force=enums/issue_status.yaml to overwrite (will lose any
      project-specific values), or merge manually.

Migrate complete: 3 added, 7 unchanged, 1 drifted.
```

#### C5. Code

- `src/tripwire/cli/migrate.py` ‚Äî NEW. Click command + orchestration.
- `src/tripwire/core/template_stamping.py` ‚Äî NEW. Shared utility:
  `stamp_into(target_dir: Path, *, dry_run: bool, force: set[str]) ->

### D. Replace `lint/issue_body_orphan_concepts` with `lint/concept_drift`

**Problem.** Today's rule
(`src/tripwire/core/lint_rules/orphan_concepts.py`) is a regex +
stopword heuristic: match any title-case word, skip a 35-entry
stopword list, flag the rest as "missing concept node". On a
normal-sized project it produces ~2,251 warnings ‚Äî every markdown
header (`## Context`, `## Implements`, `## Acceptance criteria`),
every section-starting word, every common English term. The rule's
docstring acknowledges "false positives are expected" but the CLI
exits 1 on warnings, so CI fails on every PR. Operators learn to
ignore lint output. Signal becomes noise.

**Solution.** Hard-replace with a composite-signal rule that combines
multiple weak signals into precise findings. Keep the underlying
intent ("flag load-bearing domain concepts that should be nodes but
aren't") but use signals that have meaningfully better precision.

#### D1. New rule: `lint/concept_drift`

**File:** `src/tripwire/core/lint_rules/concept_drift.py` (replaces
`orphan_concepts.py` ‚Äî delete the old file).

**Pipeline:**

1. **Strip noise zones** ‚Äî preserve only "real prose":
   - Skip markdown headers (lines starting with `#`)
   - Skip fenced code blocks (` ``` `)
   - Skip inline code (` `‚Ä¶` `)
   - Skip `[[ref]]` content (already linked)
   - Skip YAML frontmatter
   - Skip checklist scaffolding (`- [ ]`, `- [x]`)
   - Skip URLs

2. **Extract candidates from the remaining prose**:
   - Multi-word title-case noun phrases (`WebSocket Hub`, `OAuth Token`)
   - kebab-case identifiers (`project-shell`, `task-checklist`) ‚Äî
     someone meant to write `[[project-shell]]` but forgot the brackets

3. **Score each candidate** (composite signal):
   - **Cross-issue frequency**: appears in 2+ issues ‚Üí strong
   - **Within-issue frequency**: 3+ occurrences in one issue ‚Üí medium
   - **Specificity**: multi-word > single-word; kebab-case = strong
     (almost certainly a node-shaped term)

4. **Filter against existing nodes**:
   - Skip anything matching a node id, name, or alias
   - Skip anything matching a node via fuzzy match (e.g. "project
     shell" ‚Üí existing `[[project-shell]]`)
   - Skip anything in the project-level `lint/concept-allowlist.yaml`
     (item D2)

5. **Rank + report**:
   - Capped at 10 findings per issue (avoid noise floods)
   - Severity: warning (always heuristic)
   - Each finding includes: term, where it appears, occurrence count,
     inferred reason (cross-cutting / repeated / kebab-case-likely-
     ref), and fix hint

#### D2. Project-level ignorelist

**File:** `<project-dir>/lint/concept-allowlist.yaml` (new convention,
optional).

```yaml
# Terms the lint rule should treat as "intentionally not a node".
# Each entry needs a short justification ‚Äî future maintainers should
# be able to tell why something was added.
allowlist:
  - term: "Stripe"
    reason: "Mentioned once in passing as third-party reference; not
             a load-bearing concept."
  - term: "JSON"
    reason: "Format name; appears across many issues but doesn't
             warrant a node."
```

The rule loads this file at run time. Missing file = empty
allowlist (no error). The reason field is required (parser rejects
entries without it) ‚Äî forces explicit thinking instead of silent
suppression.

#### D3. Migration path for existing projects

The old rule code is deleted in the same commit as the new one.
Existing projects that ran `tripwire lint scoping` and got the noisy
output will, post-migrate, get the precise output. No data migration
needed (the rule has no persistent state).

#### D4. CI integration

The new rule's exit-code semantics stay: warnings are warnings (exit
1), errors are errors (exit 2). With the precision improvement, real
findings are now actionable instead of dismissible. The existing
project-init CI workflow template
(`.github/workflows/tripwire.yml.j2`) keeps `tripwire lint scoping`
as a gate; with the precision fix, that gate becomes meaningful.

#### D5. Code

- `src/tripwire/core/lint_rules/concept_drift.py` ‚Äî NEW (replaces
  `orphan_concepts.py`, ~150 LOC).
- `src/tripwire/core/lint_rules/__init__.py` ‚Äî drop import of
  `orphan_concepts`, add import of `concept_drift`.
- `src/tripwire/core/lint_allowlist.py` ‚Äî NEW. Loads + validates the
  per-project allowlist YAML. ~30 LOC.
- `src/tripwire/core/lint_rules/orphan_concepts.py` ‚Äî DELETE.

Optional dep: `mistune` for markdown parsing (small, stable). If
declining the dep, hand-roll a minimal markdown noise-stripper ‚Äî
acceptable since the rule only needs section-header / code-block /
fence detection, not full AST.

#### D6. Tests

- `tests/unit/test_concept_drift.py` ‚Äî NEW. Cases:
  - Section header `## Context` ‚Üí no finding (header zone stripped)
  - Code block contents ‚Üí no finding (code zone stripped)
  - `[[existing-node]]` ‚Üí no finding (already linked)
  - "WebSocket Hub" mentioned in 3 issues ‚Üí finding with frequency
    signal
  - "kanban-board" in prose ‚Üí finding marked "missing brackets,
    likely meant to be `[[kanban-board]]`"
  - Single mention of "Stripe" ‚Üí no finding (not enough signal)
  - Term in allowlist ‚Üí no finding (suppressed)
  - Allowlist entry with empty `reason` ‚Üí parser error
- `tests/unit/test_lint_allowlist.py` ‚Äî NEW. Schema validation.

#### D7. Calibration target

On `tripwire-v0` (which today produces 2,251 warnings under the old
rule), the new rule should produce: ‚â§30 findings, with ‚â•80%
precision (i.e. of the flagged items, ‚â•80% are real concepts that
would benefit from a node).

### E. CI action version policy

**Already shipped (PR #28 merged 2026-04-25):** `actions/checkout@v4`
‚Üí `@v6`, `actions/setup-node@v4` ‚Üí `@v6`, `astral-sh/setup-uv@v3` ‚Üí

#### E1. Update the init template

**File:** `src/tripwire/templates/project/.github/workflows/tripwire.yml.j2`

Bump the same versions:

```yaml
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: astral-sh/setup-uv@v8.1.0
        with:
          python-version: "3.13"
      - name: Install tripwire
        run: uv tool install tripwire-pm=={{ tripwire_version }}
      ‚Ä¶
```

Pin setup-uv to a point release because v8 dropped the floating
major tag for security per the v8.0.0 immutable-tag policy.

#### E2. Bump policy

Add a paragraph to `WORKFLOWS_REVIEW.md` (PM-skill reference)
documenting the convention:

```
**GitHub Action versions in templates.** Pin major tags
(`@v6`, `@v8.1.0`) ‚Äî never `@main` or unpinned `@v8` (the latter
fails for actions that drop the floating major tag). When a major
ships, bump in the init template AND fan out via `tripwire migrate`
to existing projects (since `migrate` would offer to re-stamp the
workflow with the new pin).
```

#### E3. No code, no tests

This is a one-line template bump + a docs paragraph. Item C's
`migrate` command is what makes the bump actually fan out to
existing projects.

## 3. Out of scope (deliberately)

- **Session.yaml schema migrations.** Item C is template re-stamping
  only. A real schema migration framework needs a `_schema_version`
  field and per-version up/down migrations; v0.8+.
- **Issue body migrations** (mass field rename, etc.).
- **`tripwire migrate --force-all`** ‚Äî too easy to clobber
  customisation. Per-path force only.
- **Auto-rewriting customised files to match templates.** Drift
  detection only. Operator decides.
- **LLM-assisted concept node suggestion** for D's findings. The rule
  flags candidates; the operator (or a v0.8 spec item) does the
  authoring.
- **GitHub App / OAuth flow** for item A's auth. PAT via env var or
  `gh auth login` is enough for v0.7.6.

   pushed, (d) `project.yaml.project_repo_url` records the SSH URL,
   (e) `enums/session_status.yaml` contains `executing` + `done`, (f)
   `.github/workflows/tripwire.yml` uses `actions/checkout@v6` etc.
2. **`--no-github-repo` skips API calls.**
3. **`--no-push` arms remote without pushing.** `git ls-remote origin
   main` returns empty.
4. **Missing token fails fast.** Without `GITHUB_TOKEN` /
   `GH_TOKEN` / gh hosts.yml, init exits non-zero with the
   documented error string.
5. **Canonical enum in fresh project.** Spawn a session in a
   `tripwire init`'d project (post-v0.7.6) ‚Äî `tripwire validate
   --strict` exits 0 on the very first session (no `enum/session_status`
   error).
6. **`tripwire migrate` is idempotent.** Run twice in a row on
   `tripwire-v0`; the second run reports "0 added, N unchanged".
7. **`tripwire migrate` reports drift.** Hand-edit a stamped file in a
   test project; `tripwire migrate --show-diffs` reports it as
   drifted but doesn't modify it.
8. **`tripwire migrate --force <path>` overwrites selectively.** Only
   the listed paths are reset; other drifted files remain.
9. **`tripwire migrate` retrofits `tripwire-v0`** ‚Äî running it picks
   up the missing CI workflow (today added by hand) and the canonical
   enum delta. Manual validation that the post-migrate state matches
   what we hand-built.
10. **`lint/concept_drift` precision.** On `tripwire-v0`'s 77-issue

11. **Allowlist suppresses cleanly.** Add an entry to
    `lint/concept-allowlist.yaml` for a flagged term; re-run; finding
    is gone.
12. **Allowlist requires `reason`.** Entry without `reason` raises a
    parser error at lint-load time.
13. **Full suite.** `uv run python -m pytest tests/ -q` ‚Äî baseline +

## 7. Risks / open questions for the project agent

- **`gh` config file format.** `~/.config/gh/hosts.yml` schema might
  vary across `gh` versions. Worth a defensive try/except + clear
  fallback message.
- **GitHub API rate limit.** Authenticated REST calls get 5,000/hr.
  v0.7.6's per-init API surface is small (1 GET, 1 POST). Far under.
- **Default visibility on auto-created repos.** Spec says `private:
  true`. For OSS-by-default orgs this is wrong; revisit if usage
  shows operators consistently passing `--public`. Recommend keeping
  private-default ‚Äî project-tracking repos contain raw plans /
  decisions / agent transcripts.
- **Migrate's drift report could be noisy** for projects with heavy
  customisation. Consider a `--ignore-drift <path>` flag if usage
  shows it. Defer until evidence.
- **Concept-drift calibration depends on project size.** The "‚â§30
  findings on 77 issues" target is calibrated to `tripwire-v0`; a
  100-issue project may legitimately have 50 real concept findings.
  The frequency thresholds (2+ issues, 3+ within-issue) are tunable;
  defaults are starting points.
- **`mistune` dependency** for the noise-zone stripper. If we want to
  avoid the dep, hand-roll the markdown stripper ‚Äî acceptable since
  we only need section-header / code-block / fence detection, not
  full AST. Recommend hand-rolling to keep the dep tree minimal.
- **Template-stamping race with manual edits.** If an operator edits
  a stamped file at the same time `tripwire migrate` runs, the diff
  could be wrong. Acceptable ‚Äî operator-induced; not in scope to
  defend against.
- **Allowlist scope.** Per-project file is the simple choice. Could
  also live in `~/.config/tripwire/concept-allowlist.yaml` for
  user-global terms ("JSON", "HTML"). Defer until usage shows the
  per-project file gets the same entries repeatedly.
- **Should `tripwire migrate` also stamp `.gitignore` updates?**
  Init writes a `.gitignore`; if the template grows entries (e.g.
  `pm-review.md` shouldn't be ignored), migrate should add them. In
  scope; the stamping logic can handle .gitignore as long as the
  diff-based add-only semantic is preserved (append to existing
  .gitignore rather than overwriting).

## 8. Why this matters

Three small failures compound into a credibility gap for new
projects:

1. **Init produces a half-working project.** Operator runs `tripwire
   init`, runs the first session, hits enum errors and missing-CI
   noise immediately. First impression: "tripwire is fragile."
2. **No retrofit path.** Once a project exists, template improvements
   never reach it. Every project becomes a snapshot of whatever
   template version it was init'd against. Drift compounds silently.
3. **Lint output is noise.** The most-prominent CI signal trains
   operators to ignore lint. Future rule additions inherit the
   credibility deficit.

v0.7.6 closes all three: init produces a working project, migrate
keeps existing projects current, and the new lint rule is precise
enough that ignoring it would be a real cost. The CI version bump is
a small concrete instance of "templates evolve; existing projects
need a fan-out path" ‚Äî making the migrate command's value tangible
on day one.
