# Keel v0.6b — Workspace migration runbook

This doc explains how to adopt the v0.6b workspace primitive in an
existing Seido setup. It is a one-shot procedure per workspace; future
projects just use `keel init --workspace`.

## Scope

- Create a new workspace repo at `~/Code/seido/projects/seido-workspace`
- Link the three existing projects (`project-kb-pivot`,
  `project-graph-ui-v2`, `project-keel-ui-init`) to it
- Identify + promote shared concepts where they exist in multiple projects

Existing projects without a `workspace:` pointer continue to work
unchanged — v0.6b is additive.

## Prerequisites

- keel v0.6b installed (or editable install pointing at this repo)
- All three test projects pass `keel validate --strict` on current state
- Clean working tree in each project repo

## Step 1: create the workspace

```bash
cd ~/Code/seido/projects
mkdir seido-workspace
cd seido-workspace

uv run --project ~/Code/seido/projects/keel keel workspace init \
    --name "Seido" --slug "seido" \
    --description "Shared concepts across Seido projects."
```

This creates `workspace.yaml`, the `nodes/` directory, and runs `git
init`.

## Step 2: link each project

From each project repo:

```bash
cd ~/Code/seido/projects/project-kb-pivot
uv run --project ~/Code/seido/projects/keel keel workspace link \
    ../seido-workspace --slug kbp

cd ~/Code/seido/projects/project-graph-ui-v2
uv run --project ~/Code/seido/projects/keel keel workspace link \
    ../seido-workspace --slug gui

cd ~/Code/seido/projects/project-keel-ui-init
uv run --project ~/Code/seido/projects/keel keel workspace link \
    ../seido-workspace --slug kui
```

Each link writes `workspace:` to the project's `project.yaml` and
appends a project entry to `workspace.yaml`.

Commit the change in each project repo:

```bash
git add project.yaml
git commit -m "chore: link to seido-workspace (v0.6b)"
```

Commit the registry updates in the workspace repo:

```bash
cd ~/Code/seido/projects/seido-workspace
git add workspace.yaml
git commit -m "chore: register three projects"
```

## Step 3: identify shared concepts (manual audit)

Shared concepts are nodes that exist with the same `id` or equivalent
meaning across two or more of the three projects. Common candidates
from the three current projects:

- Core data-model concepts (user, session, issue, etc.)
- Infrastructure concepts (auth, storage, queue, etc.)
- Cross-project protocols (event schema, API contract, etc.)

Run a quick diff to find overlapping node ids:

```bash
for p in ~/Code/seido/projects/project-*; do
    ls "$p/nodes/" 2>/dev/null | sort
done | sort | uniq -c | sort -rn | head -30
```

For each candidate node, pick the canonical version (usually the
best-documented), promote it from its home project, and fork or
delete the duplicate in the others.

## Step 4: promote canonical versions

For each canonical node:

```bash
cd ~/Code/seido/projects/<canonical-project>
uv run --project ~/Code/seido/projects/keel keel workspace promote <node-id>
```

This flips the node's `scope` to `workspace`, pushes it up, and stamps
the project copy with origin=workspace + workspace_sha.

## Step 5: sync other projects

For each other project that had a local copy of the now-canonical
node:

Option A — delete local copy + copy from workspace (fresh start):

```bash
cd ~/Code/seido/projects/project-<other>
rm nodes/<node-id>.yaml
uv run --project ~/Code/seido/projects/keel keel workspace copy <node-id>
```

Option B — pull from workspace (if the local node is identical or
close enough to auto-merge):

```bash
# Mark the local node as workspace-origin manually first by editing
# the frontmatter to add: origin: workspace, scope: workspace,
# workspace_sha: <the sha promote just created>
# Then:
uv run --project ~/Code/seido/projects/keel keel workspace pull --nodes <node-id>
```

If pull produces a merge brief, resolve via `/pm-project-sync`.

## Step 6: verify

In each project:

```bash
uv run --project ~/Code/seido/projects/keel keel validate --strict
uv run --project ~/Code/seido/projects/keel keel lint scoping
```

Expected:
- `validate --strict` passes (no errors; warnings may exist from
  pre-existing quality heuristics)
- `lint scoping` shows zero `lint/stale_workspace_nodes` findings
  (everyone on the latest workspace HEAD)
- `lint scoping` may show `lint/unpushed_promotion_candidates` (info
  or warning) for nodes still marked scope=workspace but not yet
  pushed — resolve by pushing or flipping scope back to local

Commit all three project repos' updated node files:

```bash
cd ~/Code/seido/projects/project-<name>
git add nodes/
git commit -m "chore: adopt workspace-origin shared nodes (v0.6b)"
```

## Rollback

To undo workspace adoption for a specific project:

```bash
cd ~/Code/seido/projects/project-<name>
uv run --project ~/Code/seido/projects/keel keel workspace unlink
# Node files keep their workspace-origin frontmatter — manually
# flip back to origin=local, scope=local if you want a truly
# clean standalone state.
```

To remove the workspace entirely, delete the `seido-workspace`
directory. Projects remain valid and keep their last-pulled node
copies (just without the sync link).

## Non-goals (post-migration work)

These are separate from migration and can be done on demand later:

- Building up the workspace with project-specific shared concepts
  that weren't previously cross-project
- Setting up a workspace review process before push (deferred
  Workspace manager skill)
- Remote workspace URLs (v0.6b is local-path-only)
